# Research Handbook

这份文档定义 `polymarket-ltf` 的研究方法、数据标准和策略准入要求。  
它是整个仓库里最重要的研究侧文档，面向的是：

- 量化研究
- 高频套利分析
- 数据标准化
- 回测与结果评估
- 策略进入实现前的审查

## 1. 标准研究闭环

本项目的标准闭环如下：

```text
数据获取
  -> 数据标准化
  -> 特征构造与 snapshot
  -> 回测与研究评估
  -> 策略定义与实现
  -> 执行候选 / 执行
  -> 结果输出、日志归档与复盘
```

当前仓库已经较完整支持：

- 数据获取
- snapshot 研究输入
- 回测与研究评估

当前尚未完整实现：

- 执行层
- 风控层
- 统一结果文件和日志中心

## 2. 量化与高频套利关键要素

### 2.1 数据质量

量化研究的第一性问题不是收益率，而是数据质量。

必须持续确认：

- 数据是否缺失
- 时间是否对齐
- 字段是否稳定
- 不同数据源是否存在口径冲突
- 特征是否存在未来函数或泄漏

### 2.2 Fair Value

在这个项目里，策略研究必须先回答“公允价是什么”。

当前常见 fair value 候选包括：

- Binance 中间价
- Chainlink 锚定价
- 多交易所综合参考价
- Polymarket 二元结构推导出的隐含概率

没有 fair value，就没有“偏离”和“套利机会”的定义。

### 2.3 多源对齐

Polymarket、CEX 和 oracle 的更新节奏不同，字段结构也不同。  
如果不能在同一研究时间轴上对齐，回测结果通常没有实际意义。

### 2.4 微观结构

高频套利的关键经常不在大周期方向，而在盘口细节：

- spread 是否真实可成交
- 买一卖一深度是否足够
- 可成交量是否支撑策略规模
- 部分成交概率有多高
- adverse selection 风险是否可控

### 2.5 延迟

高频套利必须回答：

- 参考价相对 Polymarket 是领先还是滞后
- 数据处理链路增加了多少延迟
- 信号生成到下单之间间隔多久
- 机会窗口是否长于你的行动时间

在本项目里，延迟问题不仅属于执行层，也属于研究层。  
如果回测假设隐含“你总能在观察到偏离后立刻行动”，那么这个回测口径本身就不专业。

### 2.6 成本

任何研究结论都必须显式考虑：

- 手续费
- 滑点
- spread crossing 成本
- 部分成交后的残余库存成本
- 跨 venue 切换和资金占用成本

### 2.7 风险与库存

高频套利不是无风险。

必须明确：

- 单 market 暴露
- 单方向库存
- 部分成交后的残余腿
- 异常行情与极端波动下的停止条件

### 2.8 可观测性与可追溯性

没有完整结果和日志，就无法判断问题到底出在：

- 数据
- 信号
- 成本
- 执行
- 风控
- 市场 regime

## 3. 核心研究问题

围绕 Polymarket crypto 市场，最关键的问题集中在下面几类。

### 3.1 Polymarket 的价格到底和什么比较

可能的比较对象包括：

- Binance 中间价
- Chainlink 锚定价
- 多 CEX 综合 fair value
- Polymarket `up/down` 结构本身

### 3.2 偏离是真机会还是噪声

需要区分：

- 结构性偏离
- 流动性空洞造成的假偏离
- 事件驱动导致的 regime shift
- 数据不同步造成的假信号

### 3.3 机会窗口是否足够长

如果价差机会持续时间短于你的链路延迟，再漂亮的回测也不具备执行价值。

### 3.4 二元市场结构如何影响研究与执行

Polymarket 不是传统现货市场。必须考虑：

- `up/down` 是互补腿
- 盘口不一定对称
- 买 `down` 不等于抽象意义上的“做空 up”
- 到期 payoff 与中途平仓逻辑不同

### 3.5 扣除真实成本后是否仍然成立

研究结论不能只看方向判断是否正确，还要看：

- 扣掉费用后是否仍然为正
- 加入滑点后是否仍然稳定
- 部分成交后是否仍然成立

### 3.6 研究结论是否能转成执行规则

一个统计信号不是一个可执行策略。  
必须继续明确：

- 何时开仓
- 开哪条腿
- 开多少
- 何时平仓
- 何时不交易

### 3.7 多交易所扩展后如何保持统一标准

随着更多交易所接入，最常见的问题是：

- symbol 不统一
- 时间戳口径不统一
- book / trade / mark / oracle 混杂
- fair value 参考系漂移

## 4. 数据、结果与日志标准

### 4.1 总体原则

- 来源清晰
- 语义稳定
- 层次分离
- 结果可追溯

任何一次回测或执行结果，都应该能追溯到：

- 输入数据
- 策略版本
- 参数
- 运行时间

### 4.2 当前正式研究边界

当前实时层与研究层之间的正式接口是：

```text
data/snapshots/<symbol>/<interval>/<market_slug>.csv
```

这是当前仓库最重要的数据契约。

### 4.3 推荐目标目录

当前不需要一次性实现全部目录，但长期建议收敛到：

```text
data/
├── raw/
│   ├── polymarket/
│   ├── cex/
│   └── oracle/
├── normalized/
├── snapshots/
├── backtests/
│   ├── runs/
│   ├── reports/
│   └── exports/
├── execution/
└── metadata/

logs/
├── realtime/
├── backtest/
└── execution/
```

### 4.4 命名标准

目录命名：

- 使用 `snake_case`
- 交易所名称小写
- symbol 使用统一 slug，如 `btc`、`eth`
- interval 使用统一字符串，如 `5m`、`15m`

字段命名：

- 带清晰来源前缀，如 `binance_`、`chainlink_`、`up_`、`down_`
- 带清晰语义后缀，如 `bid_price`、`ask_price`、`mid_price`、`size`
- 避免使用模糊字段，如 `price`、`value`、`run`、`data`

### 4.5 回测结果标准

一次标准回测至少应产出：

明细结果：

- `run_id`
- `strategy_name`
- `dataset_path` 或 `dataset_id`
- `symbol`
- `interval`
- `market_slug`
- `starting_cash`
- `ending_cash`
- `ending_equity`
- `return_pct`
- `max_drawdown_pct`
- `trade_count`
- `parameter_blob`

汇总结果：

- 总文件数
- 总样本行数
- 总成交数
- 平均收益率
- 平均最大回撤
- 胜率
- 最好 / 最差 market

### 4.6 执行结果标准

未来进入执行层后，至少应具备：

- `run_id`
- `venue`
- `symbol`
- `market_slug`
- `side`
- `intended_price`
- `sent_price`
- `filled_price`
- `intended_size`
- `filled_size`
- `status`
- `reason`
- `created_at`
- `sent_at`
- `ack_at`
- `filled_at`

### 4.7 日志标准

实时日志至少记录：

- 数据源连接 / 断开
- 重连
- 市场切换
- snapshot 写入异常

回测日志至少记录：

- 数据路径
- 策略名与参数
- 回测开始与结束
- 聚合统计
- 异常文件

执行日志至少记录：

- 信号生成
- 风控拒绝
- 下单请求
- 订单确认
- 成交回报
- 重试与失败

## 5. 策略研究准入清单

任何新策略在进入实现前，至少要通过下面这份检查。

### 5.1 研究假设

- 这个策略利用的偏差是什么
- 偏差来自信息、流动性、结构还是延迟
- 为什么这个偏差可能持续存在
- 它在什么条件下失效

### 5.2 数据准备

- 需要哪些 Polymarket / CEX / oracle 数据
- 数据是否完整
- 时间是否对齐
- 当前 snapshot 字段是否足够
- 是否存在未来函数风险

### 5.3 Fair Value 与信号定义

- fair value 定义是否清楚
- 偏离如何量化
- 开仓阈值如何定义
- 平仓阈值如何定义
- 什么情况下明确不交易

### 5.4 回测口径

- 是否包含手续费
- 是否包含滑点
- 是否做了 `5m` / `15m` 分组验证
- 是否做了跨 market 验证
- 是否存在仅在少数样本有效的情况

### 5.5 执行可行性

- 机会窗口是否长于链路延迟
- 盘口深度是否足够
- 部分成交如何处理
- 残余库存如何处理
- 是否依赖理想成交假设
- 是否显式定义了延迟预算和失效窗口

### 5.6 风险控制

- 最大单笔风险是多少
- 单 market 风险上限是多少
- 单 symbol 风险上限是多少
- 连续亏损或异常行情何时停止交易

### 5.7 输出与复盘

- 是否生成 `run_id`
- 是否记录策略版本
- 是否记录参数
- 是否保存结果文件
- 是否保存运行日志
- 是否能从结果反查到输入数据

## 6. 常见研究误区

- 只看收益率，不看成交概率
- 只看中间价，不看盘口深度
- 只看方向，不看成本
- 只看单个 market 的漂亮曲线
- 只看历史拟合，不看数据质量
- 只看信号，不看延迟和库存
- 没有日志却试图解释回撤

## 7. 当前最值得优先强化的方向

对当前仓库而言，最有价值的优先级是：

1. 更稳定的多源数据接入
2. 更可信的 fair value 和 spread 特征
3. 更结构化的回测结果与参数记录
4. 更明确的策略定义与版本化
5. 为未来执行层预留结果、日志和状态接口
